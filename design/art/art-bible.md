## Visual Identity Anchor

**"深空废墟中孤独的霓虹"** —— 在近乎完全黑暗的太空背景中，机甲、武器、敌人、NPC 都用高饱和度的霓虹色点亮，每一个发光元素都意味着"这里有故事"。

**核心规则**：每一个发光的像素都必须回答一个问题："为什么这光在这里？" 如果没有答案，就让它灭掉。

**支撑原则**：
- **黑暗 + 单一光源**：每个房间只用一个主光源 + 1-2 个环境光晕
- **暖机甲 / 冷废墟**：前景角色暖橙红，背景环境冷蓝灰，立刻分离
- **像素精度优先于细节量**：32x32 像素基础单位优先于细节堆砌

**色彩哲学**：背景 90% 暗色 + 10% 高饱和点缀。和"密度"承诺一致 —— 在大面积"未知"中，少数"已知"被强烈强调。

## Section 1: Visual Identity Statement

**One-line 视觉规则**：
> **"每个发光的像素都必须回答一个问题：'为什么这光在这里？'如果没有答案，就让它灭掉。"**

这条规则直接来自 brainstorm 的 Visual Identity Anchor，并且：
- **服务于支柱 1 (探索密度)** —— 黑暗 + 单一光源 = "未知的空间"
- **服务于支柱 2 (发现 > 数值)** —— "发光的点 = 玩家要看见的东西"
- **服务于支柱 4 (真相是收集的结果)** —— 发光的 NPC 录音 = 剧情线索

### 支撑原则

#### 原则 1: 黑暗 + 单一光源
**测试**："这个房间有 5 个独立光源" → 错。改成一个主光源 + 1-2 个环境光晕。
**效果**：把玩家的注意力"逼"到光源 = 引导探索。

#### 原则 2: 暖机甲 / 冷废墟
**测试**："机甲和敌人都是冷蓝色" → 错。前景角色（机甲）用**暖橙红**做主色，背景环境用**冷蓝灰**做主色。
**效果**：玩家一眼分清"我"和"环境"——视觉层次立刻建立。

#### 原则 3: 像素精度优先于细节量
**测试**："我想给机甲加 5 个动态光影细节" → 错。优先 32x32 基础单位的清晰轮廓。
**效果**：保持"FC 时代 RPG"的视觉记忆 + 避免"想太多画不像"。

## Section 2: Mood & Atmosphere

按游戏状态定义情绪目标：

| 状态 | 主要情绪 | 灯光特征 | 氛围形容词 | 能量 |
|------|---------|---------|-----------|------|
| **探索（核心）** | 孤独的兴奋 | 暖机甲头灯 + 冷蓝灰背景 + 高对比 | 孤寂、神秘、轻微压迫 | 沉思 / 持续 |
| **暗雷遇敌** | 警惕的紧张 | 屏幕瞬间变红 + 高频闪烁 1-2 秒 | 紧绷、肾上腺素、危险 | 急促（短） |
| **战斗（手动模式）** | 战术专注 | 暖色机甲头灯常亮 + 敌人用冷色背光 | 紧张、清晰、玩家控制感 | 节奏性 |
| **战斗（自动模式）** | 减压的观看 | 灯光保持探索态 + 屏幕边缘暖色微晕 | 平稳、慵懒、舒适 | 低能量 |
| **胜利 / 战利品** | 满足的狂喜 | 暖金光短暂扫描全屏 + 像素粒子爆开 | 庆祝、爽快、有回报 | 短暂高能 |
| **剧情揭示（关键发现）** | 敬畏的发现 | 屏幕慢速聚焦到发光物 + 其他光源瞬间熄灭 | 仪式、郑重、慢 | 静止（短） |
| **图鉴 / 菜单** | 中性的查看 | 暖白底光 + 冷灰边框 | 干净、有条理、博物馆感 | 低能量 |
| **失败 / 死亡** | 沉重的失败 | 屏幕快速变黑 + 暖色机甲头灯闪烁熄灭 | 沉痛、终结感、孤独 | 短促（短） |

### 关键差异设计

- **战斗手动 vs 自动**：灯光设计要明确区分（让玩家有"我在控制 / 我在挂机"的心理提示）
- **遇敌 vs 战斗**：遇敌是 1-2 秒的"红屏冲击"，战斗是"红屏退场后的稳定态"
- **剧情揭示 vs 战利品**：揭示是"聚焦 + 静止"，战利品是"扩散 + 爆发"
- **章节色温递进**：第 1 章冷蓝灰 → 第 2 章冷紫灰 → 第 3 章暖橙红（玩家情绪随章节升温）

## Section 3: Shape Language

### 核心几何词汇

| 类别 | 主导形状 | 哲学 | 例子 |
|------|---------|------|------|
| **机甲轮廓** | 棱角 + 直线主导 | 工业感、机械感、可读 | 机甲胸甲是矩形 + 三角肩甲 |
| **敌人轮廓** | 有机 + 异形 | 与机甲形成对比，凸显"非人" | 异形是曲线 + 不规则孔洞 |
| **环境（卫星）** | 直角 + 模块化 | 强调"人造物"+"50 年前有人住过" | 走廊 = 长方形块状空间，门是矩形 |
| **环境（废墟）** | 破口 + 碎裂 | 强调"50 年没人维护" | 破损墙壁 = 不规则缺口 + 钢筋外露 |
| **UI 元素** | 圆角 + 数字科技感 | 与世界区分，UI 是"操作系统" | 图标是圆角矩形 + 内部细线条 |
| **发光物** | 几何对称 | 强调"这是人造光源" | 灯是圆形、应急灯是方形、武器光柱是直线 |

### 关键设计测试

- **缩略图可读性**："玩家在 64x64 缩略图能分清这是机甲还是敌人吗？" → 棱角 vs 有机 = 一眼可读
- **氛围塑造**："玩家能在不读图的情况下分辨'这里是安全区还是危险区'吗？" → 直角 = 安全，碎裂 = 危险
- **UI 一致性**："UI 元素和世界元素用同一种形状语言吗？" → **故意区分**：UI 是圆角（操作系统），世界是棱角（人造物）

## Section 4: Color System

### 主色板（Primary Palette）

| 角色 | 颜色名 | HEX | 含义 |
|------|--------|-----|------|
| **背景主色** | 深空蓝灰 | #0E1420 | 太空背景，90% 的视觉 |
| **机甲主色** | 暖橙红 | #FF6B3D | 玩家机甲，永远是"我" |
| **敌人主色** | 毒绿 | #4ADE80 | 危险，警惕，需要应对 |
| **隐藏物** | 电浆蓝 | #38BDF8 | 可拾取、有回报、值得捡 |
| **剧情物** | 紫红 | #A855F7 | 真相碎片、NPC 录音、关键 |
| **安全区** | 暖白 | #F5F5DC | 存档点、休息区 |
| **危险提示** | 警告红 | #DC2626 | 警告、Boss、不可逆 |

### 色彩语义规则

- **红色** = 危险（敌人攻击、警告、Boss 战、不可逆）
- **金色 / 橙** = 玩家 / 奖励 / 安全
- **蓝色** = 互动 / 拾取 / 探索奖励
- **紫色** = 剧情 / 知识 / 不可错过
- **绿色** = 危险（敌人）/ 治愈（道具，双语义需形状 backup）
- **灰色** = 背景 / 已访问 / 无价值

### 按章节色温规则

| 章节 | 背景主色温 | 情绪 |
|------|----------|------|
| **章节 1：前哨基地** | 冷蓝灰（#0E1420） | 神秘、新手 |
| **章节 2：研究设施** | 冷紫灰（#1A1525） | 紧张、阴谋 |
| **章节 3：反应堆核心** | 暖橙红（#2A1A15） | 危险、真相 |

### UI 色板（与世界色板故意区分）

- **UI 底色** = 透明黑 + 50% 不透明
- **UI 文字** = 暖白（#F5F5DC）
- **UI 边框** = 冷蓝灰（#1F2937）
- **UI 强调** = 电浆蓝（#38BDF8），与世界"拾取色"一致（建立"这是好东西"的跨场景认知）

### 色盲安全规则

- **绿色敌人 vs 绿色治疗道具** 容易混淆 → 道具加 "+" 形状 backup
- **红色危险 vs 红色 boss** 容易混淆 → Boss 用**紫红色**而非纯红
- **剧情紫 vs 敌人毒绿** 容易混淆 → 剧情物用**形状**（终端、录音）backup
- **每个语义色都必须有"形状 / 声音" backup**（不全靠颜色）

## Section 5: Character Design Direction

### 玩家角色：漫游者（The Roamer）

| 维度 | 规则 |
|------|------|
| **视觉原型** | 棱角分明的机甲 + 三角肩甲 + 矩形胸甲 + 厚腿 |
| **主色** | 暖橙红（#FF6B3D）—— 永远不能变 |
| **次色** | 冷灰蓝（#1F2937）—— 机甲骨架 |
| **头灯** | 永远开启的暖白（#F5F5DC）—— 唯一光源 |
| **眼睛/视窗** | 电浆蓝（#38BDF8）—— 表情载体 |
| **大小** | 32x32 基础单位，移动时占 1 格（8x8 网格） |
| **朝向** | 4 方向（上下左右），不用 8 方向（节省美术） |
| **攻击** | 武器从右臂发射，激光 = 直线，弹道 = 抛物线 |
| **被击** | 暖橙红闪烁 0.5 秒 + 屏幕边缘暖色微晕 |
| **移动动画** | 2 帧循环（腿交替） |

### 敌人类型视觉规则（缩略图可读性）

| 敌人类型 | 视觉原型 | 主色 |
|---------|---------|------|
| **小怪·近战** | 圆形 / 椭圆形 + 短腿 | 毒绿 |
| **小怪·远程** | 三角形 / 锥形 + 长臂 | 毒绿 |
| **中怪·飞行** | 翼形 + 圆形主体 | 毒绿 + 紫红斑纹 |
| **中怪·重型** | 矩形 + 多腿 + 厚甲 | 毒绿 + 灰黑甲 |
| **Boss** | 异形 + 不规则孔洞 + 发光核心 | 紫红色 + 警告红脉动 |

### 关键规则

- **缩略图可读性测试**："在 64x64 缩略图中，玩家能一眼分清是哪种敌人吗？" → 通过棱角/曲线/颜色斑纹的对比达成
- **不要给敌人"眼睛"** —— 敌人是"非人"的，机器感强
- **每个敌人都有"被打败时"的特殊动画** —— 不是消失，而是爆开像素粒子

### LOD 哲学

- **正常游戏距离**（屏幕 1/4 高）：完整 32x32 像素细节
- **远景 / 小地图**：缩到 16x16，移除细节，保留轮廓识别
- **大地图标记**：8x8 简化色块，仅保留主色和形状

## Section 6: Environment Design Language

### 建筑风格

| 元素 | 风格 | 理由 |
|------|------|------|
| **走廊** | 直角 + 模块化（重复矩形块） | "人造物"+" 50 年前有人住过" |
| **门** | 矩形 + 两侧对称灯柱 | 仪式感 + "打开就是进入" |
| **房间** | 多边形 + 不规则（但有几何中心） | 不希望是"盒子的盒子的盒子" |
| **管道** | 圆柱 + 90 度弯头 | 工业感 |
| **通风井** | 圆形 + 网格 | 探索通道 |
| **控制台** | 矩形屏幕 + 圆角按钮 | 操作系统感 |
| **终端日志点** | 小型浮空屏幕 + 紫红发光 | 剧情物，标识清晰 |

### 纹理哲学

- **PBR? No** —— 2D 像素艺术，**手工绘制的色块 + 1-2 层细节**
- **瓦片集（Tile Set）**：每个区域用 4-6 种瓦片
  - 基础墙（深空蓝灰）
  - 破损墙（深空蓝灰 + 不规则缺口）
  - 地板（更暗的灰）
  - 边缘（高亮 1-2 像素，提示"这里是边界"）
  - 装饰（管道、灯、警示条）
- **不用动态光影** —— 用**手绘的"伪光影"**：墙的顶部亮 1-2 像素、底部暗 1-2 像素，假装有光
- **每个房间最多 3 个独特细节** —— 不堆细节

### 道具密度规则

| 区域类型 | 道具密度 | 理由 |
|---------|---------|------|
| **走廊** | 低（每屏 1-2 个） | 快速通过，不打扰节奏 |
| **可探索房间** | 中（每屏 3-5 个） | 探索回报 |
| **关键剧情点** | 高（每屏 5-8 个） | 仪式感 |
| **Boss 战前** | 中-高（每屏 4-6 个） | 准备感 |
| **存档点** | 低（每屏 1-2 个） | 休息 |

### 环境叙事规则（"用视觉讲故事，不靠文字"）

| 视觉细节 | 叙事含义 |
|---------|---------|
| 墙上散落的工具 | "这里曾经有人工作过" |
| 应急灯的电池提示"剩 50%" | "卫星的能源在衰退" |
| 墙上的弹痕 + 干掉的血迹 | "这里发生过战斗" |
| 倒下的机甲（不是玩家的型号） | "有人没能活下来" |
| 破损的玻璃 + 窗外的星空 | "你可以看见外面" |
| 控制台上的最后一次操作（半完成） | "谁在这里？他去哪了？" |

**关键规则**：**不写明文故事** —— 让玩家从视觉细节**自己拼**。这直接服务于支柱 4（真相是收集的结果）。

## Section 7: UI / HUD Visual Direction

### 整体方向：diegetic + screen-space 混合

- **Diegetic（沉浸式 UI）**：机甲胸前的显示屏 = 机甲的"操作系统"，玩家视觉上"在机甲内部"
- **Screen-space（HUD）**：战斗中的数字、菜单、图鉴 = 操作系统弹窗

### 字体方向

- **标题字体**：单宽像素字体（类似 FC 时代"Press Start 2P"），大号、暖白
- **正文字体**：同一字体的可读尺寸（12-14px），冷灰
- **数字字体**：等宽数字（强调数值但不被注意）
- **剧情字体**：可读尺寸 + 暖白 + 慢速出现（每字 50ms）

### 图标风格

- **扁平 + 描边** —— 不用拟物，不用 3D
- **统一 16x16 基础单位** —— 缩略图可读
- **暖色 vs 冷色二元**：暖色 = 玩家用、冷色 = 系统用

### UI 动画感

- **菜单切换**：100-150ms 横向滑入
- **数字变化**：100ms 滚动（如金钱 +50 → 0 滚到 50）
- **按钮反馈**：50ms 缩放（按下时缩到 90%，松开时弹回 100%）
- **新发现提示**：屏幕中心 1.5 秒暖金光晕 + 图标浮入

### HUD 元素（战斗中屏幕边缘）

| 元素 | 位置 | 风格 |
|------|------|------|
| **HP 条** | 左上 | 暖橙红长条 + 数字 |
| **机甲部位状态** | 左上（HP 下方） | 4 个小图标（头/胸/臂/腿） |
| **当前武器** | 右上 | 武器图标 + 弹药类型 |
| **当前弹药量** | 右上（武器下方） | 数字 |
| **模式指示**（手动/自动） | 右上角 | 文字"MANUAL" / "AUTO" |
| **敌人 HP** | 敌人上方 | 红色长条 |
| **遭遇敌数** | 中上 | "X / Y" |
| **道具快捷栏** | 下中 | 4 个圆角图标 |

### 关键设计原则

- **不打扰**：HUD 元素不闪、不动（除了数字变化）—— 让玩家专注于游戏
- **不抢戏**：所有 HUD 元素都用 50% 透明底色 + 暖白文字，不和游戏世界争夺注意力
- **可隐藏**：按一个键可隐藏所有 HUD（截图用）
- **色盲友好**：所有 UI 元素都有形状 backup（HP 条 = 长方形、敌人 HP = 红色但有"X/Y"数字）

## Section 8: Asset Standards

### 文件格式

| 资产类型 | 格式 | 理由 |
|---------|------|------|
| **像素美术** | PNG（透明背景） | 通用、Godot 原生支持 |
| **字体** | WOFF2（推荐）+ Godot Bitmap Font (.font) | 可读性 + 性能 |
| **音频** | OGG Vorbis | 体积小、Godot 原生支持 |
| **音乐** | OGG Vorbis | 同上 |
| **数据** | JSON / .tres（Godot 资源） | 配置可外部修改 |
| **关卡** | .tscn（Godot 场景） | Godot 原生 |

### 命名规范（GDScript 风格）

```
# 命名模板
<asset_type>_<subject>_<variant>_<size>.<ext>

# 例子
char_player_roamer_idle_32x32.png
char_enemy_small_melee_walk_32x32.png
weapon_laser_basic_32x32.png
tile_wall_corridor_32x32.png
ui_icon_health_16x16.png
sfx_combat_hit_impact.ogg
bgm_chapter1_exploration.ogg
```

### 纹理分辨率分层

| 资产 | 基础单位 | 美术放大倍数 | 实际像素 |
|------|---------|------------|---------|
| **角色** | 32x32 | 2x | 64x64 渲染 |
| **小怪** | 32x32 | 2x | 64x64 渲染 |
| **Boss** | 64x64 | 2x | 128x128 渲染 |
| **瓦片** | 32x32 | 1x（直接用） | 32x32 |
| **UI 图标** | 16x16 | 2x | 32x32 渲染 |
| **大背景** | 1920x1080 | 1x | 全屏 |
| **粒子** | 8x8 - 16x16 | 1-2x | 小 |

### LOD 等级

| 资产 | 近距离 | 中距离 | 远距离 |
|------|-------|-------|-------|
| **角色** | 32x32 全细节 | 32x32 完整 | 16x16 简化色块 |
| **敌人** | 32x32 全细节 | 32x32 完整 | 16x16 简化轮廓 |
| **瓦片** | 32x32 | 32x32 | 32x32（瓦片统一） |

### 导出设置

- **PNG**：无损、不压缩、Alpha 通道
- **音频**：OGG q=5（音乐 q=4，因为长）
- **JSON**：缩进 2 空格、UTF-8
- **场景 (.tscn)**：版本控制友好格式（Godot 4.x 默认）

### 性能预算（来自 technical-preferences.md）

- **目标帧率**：60 FPS
- **帧预算**：16.6ms
- **Draw Calls**：~200/场景
- **内存天花板**：500MB

### 实际生产约束

- **单场景不超过 50 个角色**（避免 draw call 爆炸）
- **单场景不超过 200 个粒子**（避免粒子系统过载）
- **不使用动态光影**（改用手绘伪光影）—— 这是 2D 像素 + 性能的关键决策
- **不使用 bloom / glow 屏幕后处理**（用精灵发光替代，draw call 友好）
- **瓦片集限制**：每章节最多 6 种瓦片类型
- **粒子集限制**：全游戏最多 4-5 套粒子预设（爆炸、电浆、烟、修复、剧情发光）

### 冲突解决（艺术 vs 技术）

- **理想** = 4K 纹理 + 复杂粒子 + 动态光影
- **约束后** = 32x32 基础 + 简单粒子 + 手绘伪光影
- **取舍理由**：2D 像素 + 60FPS + 几周 ship 的约束下，"美术清晰度" > "物理正确性"

## Section 9: Reference Direction

### 参考方向

| 参考源 | 借鉴什么 | 避开什么 | 为什么这个参考 |
|--------|---------|---------|--------------|
| **《重装机兵》（FC, 1991）** | 暗雷遇敌节奏、回合制菜单 UI、战车图鉴的"完成度"激励、像素机甲的轮廓识别 | 现代 UI 元素（图标、详细数字）、复杂的机甲设计 | 这是游戏的**情感根基** —— 玩家要找的是这个感觉 |
| **《Into the Breach》（Subset Games, 2018）** | 紧凑的战术回合制、build 驱动的难度设计、auto-battle 选项的友好性 | 完全现代的极简 UI、3D 渲染、复杂技能树 | 验证"build 深度 + 紧凑战斗"的市场 |
| **《Outer Wilds》（Mobius Digital, 2019）** | 收集驱动叙事、用环境细节代替文字、太空的孤独美学、视觉上"每个细节都重要" | 3D 写实风格、复杂的航天器控制、动态光照 | 验证"探索 + 收集 + 太空"的黄金配比 |
| **《密特罗德》（FC, 1986）** | 2D 太空像素美学、隐藏区域的视觉提示（"看起来不一样就是有东西"）、迷宫探索的密度感 | 平台跳跃、动作射击、复杂的地图系统 | 验证"2D 太空像素"的视觉基底 |
| **《极乐迪斯科》（ZA/UM, 2019）** | 紧凑叙事、收集的"考古感"、把"找东西"做得有仪式感、画面配色（紫红 / 暖白 / 冷蓝） | 油画风格、写实人物、复杂对话树 | 验证"色彩哲学"和"收集仪式感" |

### Non-game 参考

| 来源 | 借鉴什么 | 避开什么 |
|------|---------|---------|
| **《异形》（Alien, 1979）** | 太空船内部的压抑感、"低光 + 单一光源"的视觉压迫、环境的"工业感" | 真实的太空服设计、复杂的生物机械 |
| **宫崎骏《天空之城》** | 失落的科技文明的视觉感、废墟的诗意、人造物的怀旧 | 写实人物、温暖的色调（我们要冷色为主） |
| **《银翼杀手》（1982）** | 太空站的内部美学、霓虹 + 暗色的强烈对比、"雨夜 + 暖色广告牌"的光感 | 写实人物、过度的全息广告 |

### 加成规则（如何综合）

> **不要"试图复刻 X"** —— 综合这些参考的**核心技法**（暗雷 = 重装机兵，build 深度 = Into the Breach，收集 = Outer Wilds，色彩 = 极乐迪斯科，2D = 密特罗德），但**视觉输出应该是 Railhunter 自己**。

### 关键警告

- **不要做"另一个《重装机兵》"** —— 用回合制但要现代化（时机感、build 深度、剧情节奏）
- **不要做"另一个《异形》"** —— 用压抑感但要有"暖机甲"作为玩家救赎
- **不要做"另一个《极乐迪斯科》"** —— 用收集仪式感但不要写几小时文本（首次开发 + 几周时间）
- **不要做"另一个《密特罗德》"** —— 用 2D 像素美学但要 RPG 系统，不要纯平台动作
