# ADR-0009: Input Binding Strategy

## Status

Accepted

## Date

2026-06-12

## Last Verified

2026-06-12

## Decision Makers

User + técnico-director (self-review)

## Summary

Railhunter uses a **47-action Godot `InputMap` defined in `design/registry/input-bindings.yaml`** as the single source of truth, with the YAML being the canonical store and `project.godot` `[input]` section being the runtime artifact generated by a CI sync step. The 47 actions are **closed-set** (any new action requires updating the GDD + binding registry). Device hot-swap behavior (SDL3 backend, 4.5+ HIGH RISK) is defined: same physical keybind hot-swap is supported, but visual focus routing is separate from action dispatch (4.6 dual-focus model). Codifies `player-input.md` G1 + E1 + E3 (revised).

## Context

### Problem Statement

`player-input.md` declares 47 input actions grouped by domain:
- Exploration: 8
- Battle — Combat: 12
- Battle — Build: 4
- Battle — Mode: 2
- Menu / UI: 8
- Terminal / Codex: 4
- System: 5
- Debug (dev build only): 4
- **Total: 47**

This is a **closed set** — any new action requires updating the GDD. But the 47 actions must live somewhere Godot can read. The options are:

1. **Hard-code in `project.godot`** — devs edit by hand, no source of truth outside Godot
2. **Single YAML file** — `design/registry/input-bindings.yaml`, parsed by a CI step that generates `project.godot`
3. **Multiple YAML files** — one per domain (exploration.yaml, battle.yaml, etc.) — over-engineered for 47 actions

The chosen approach is **(2)**: single YAML, single source of truth, generated `project.godot` via CI.

Additionally, the 47 actions must be **documented** — devs need to know which physical key/button does what. The YAML includes the binding map per action (per `player-input.md` G1 summary table).

### Current State

- `design/registry/input-bindings.yaml` already exists (per session state files created in prior GDD work)
- `player-input.md` G1 declares 47 actions + per-domain binding summary
- `architecture.md` §4a InputBus: "47 actions (see `design/registry/input-bindings.yaml`)"
- `architecture.md` §4a HIGH RISK: "dual-focus 4.6 — `Input.is_action_just_pressed` covers both mouse/touch and keyboard/gamepad, but visual focus (HUD highlighting) is now separate. `grab_focus()` only affects keyboard/gamepad."
- The YAML → `project.godot` sync step is **not yet specified**

### Constraints

- **47 actions closed set** — adding a 48th requires GDD update + binding registry update + linter pass
- **Godot 4.6 dual-focus** — action dispatch ≠ visual focus; must be designed around
- **SDL3 gamepad backend** (4.5+) — hot-swap behavior may differ from pre-4.5
- **Solo dev** — YAML editing must be one tool, not multiple
- **MVP scope** — no rebinding UI (per player-input.md C-R7)

### Requirements

- **Single source of truth** — YAML is canonical, `project.godot` is generated
- **Action names = StringName** — not strings (per `player-input.md` C-R2)
- **Closed set** — 47 actions, linter enforced
- **Per-domain binding summary** — devs can read YAML and know "A = attack, B = defend"
- **CI sync step** — generates `project.godot` from YAML at build time
- **Device hot-swap** — same physical keybind hot-swaps between devices (e.g., unplug gamepad, plug different one)
- **Visual focus routing separate from action dispatch** (4.6 dual-focus)
- **Debug actions stripped from release build** (per `player-input.md` C-R1)
- **Determinism** — bindings are the same for every player in every playthrough (no per-player rebinding for MVP)

### Architecture

```

  │                                                            │
  │  Source of truth. 47 actions.                              │
  │  Each action:                                              │
```

```gdscript
# === CI sync step ===
# File: tools/sync_input_bindings.py
# Reads design/registry/input-bindings.yaml
# Generates project.godot [input] section
# Idempotent

import yaml

INPUT_BINDINGS_PATH = "design/registry/input-bindings.yaml"
PROJECT_GODOT_PATH = "project.godot"

def main():
    with open(INPUT_BINDINGS_PATH) as f:
        bindings = yaml.safe_load(f)
    
    actions = bindings["actions"]
    
    # Validate 47 actions
    if len(actions) != 47:
        raise ValueError(f"Expected 47 actions, got {len(actions)}")
    
    # Read project.godot
    with open(PROJECT_GODOT_PATH) as f:
        lines = f.readlines()
    
    # Find [input] section
    in_input = False
    new_lines = []
    for line in lines:
        if line.strip() == "[input]":
            in_input = True
            new_lines.append(line)
            # Generate input entries
            for action in actions:
                new_lines.extend(generate_input_entry(action))
            continue
        if in_input and line.strip().startswith("["):
            in_input = False
        if not in_input:
            new_lines.append(line)
    
    # Write back
    with open(PROJECT_GODOT_PATH, "w") as f:
        f.writelines(new_lines)

def generate_input_entry(action):
    """Generate Godot project.godot [input] entry for one action."""
    entries = []
    entries.append(f'\n')
    # Keyboard
    for kb in action.get("keyboard", []):
        modifiers = " + ".join(kb.get("modifiers", []))
        if modifiers:
            entries.append(f'{action["name"]}={{"deadzone": 0.5, "events": [Object(InputEventKey,"resource_local_to_scene": false, "resource_name": "", "device": -1, "window_id": 0, "alt_pressed": false, "shift_pressed": {("true" if "SHIFT" in modifiers else "false")}, "ctrl_pressed": {("true" if "CTRL" in modifiers else "false")}, "meta_pressed": false, "pressed": true, "keycode": 0, "physical_keycode": {key_to_code(kb["key"])}, "key_label": 0, "unicode": 0, "location": 0, "echo": false, "script": null)]}}\n')
    # ... gamepad, mouse ...
    return entries
```

```gdscript
# === InputBus (autoload #2) — runtime assertion ===
# File: src/autoload/input_bus.gd (excerpt)
class_name InputBus
extends Node

const EXPECTED_ACTION_COUNT: int = 47
const EXPECTED_ACTIONS: Array[StringName] = [
    # Exploration
    &"move_up", &"move_down", &"move_left", &"move_right",
    &"interact", &"dash", &"open_codex", &"open_minimap",
    # Battle - Combat
    &"attack_primary", &"attack_secondary", &"use_item",
    &"defend", &"skip_turn", &"target_next", &"target_prev",
    &"cycle_weapons", &"confirm_target", &"cancel_target",
    &"flee_battle", &"pause",
    # Battle - Build
    &"weapon_slot_1", &"weapon_slot_2", &"weapon_slot_3",
    &"cycle_ammo",
    # Battle - Mode
    &"toggle_mode", &"toggle_auto_attack",
    # Menu / UI
    &"menu_up", &"menu_down", &"menu_left", &"menu_right",
    &"menu_confirm", &"menu_cancel", &"menu_tab_next", &"menu_tab_prev",
    # Terminal / Codex
    &"play_log", &"skip_log", &"codex_filter", &"codex_close",
    # System
    &"pause", &"quick_save", &"quick_load", &"screenshot", &"quit_to_title",
    # Debug
    &"debug_toggle_god", &"debug_warp", &"debug_fill_codex", &"debug_damage",
]

func _ready() -> void:
    var actions: Array[StringName] = InputMap.get_actions()
    
    # Validate count
    if actions.size() != EXPECTED_ACTION_COUNT:
        push_error("InputBus: expected %d actions, got %d" % [EXPECTED_ACTION_COUNT, actions.size()])
    
    # Validate each expected action exists
    for expected in EXPECTED_ACTIONS:
        if not InputMap.has_action(expected):
            push_error("InputBus: missing action %s" % expected)
    
    # Validate no extra actions (closed set)
    for actual in actions:
        if actual not in EXPECTED_ACTIONS:
            push_warning("InputBus: unexpected action %s (not in EXPECTED_ACTIONS)" % actual)
    
    set_process_input(true)
```

### Implementation Guidelines

#### YAML schema

```yaml
# Top-level structure
version: 1                       # schema version of the YAML
actions:                        # Array of action definitions
  - name: move_up
    domain: EXPLORATION
    # ... fields ...

# Domains (closed enum):
# - EXPLORATION       (8 actions)
# - BATTLE_COMBAT     (12 actions)
# - BATTLE_BUILD      (4 actions)
# - BATTLE_MODE       (2 actions)
# - MENU              (8 actions)
# - TERMINAL          (4 actions)
# - CODEX             (4 actions, same as TERMINAL in player-input.md — but distinguished for documentation)
# - SYSTEM            (5 actions)
# - DEBUG             (4 actions, stripped from release build)
```

#### Domain count vs action count

Per `player-input.md` G1:
- 8 (Exploration) + 12 (Battle Combat) + 4 (Battle Build) + 2 (Battle Mode) + 8 (Menu/UI) + 4 (Terminal/Codex) + 5 (System) + 4 (Debug) = **47 actions**

The 47 must match between:
- `player-input.md` G1 (文档)
- `input-bindings.yaml` (canonical store)
- `EXPECTED_ACTIONS` in InputBus (runtime check)
- `project.godot` [input] section (generated artifact)

Linter asserts all 4 match. If any one is out of sync, CI fails.

#### CI sync step

- **When**: pre-commit (local fast check) + CI build (definitive)
- **What it does**: regenerates `project.godot [input]` from YAML
- **Idempotent**: running twice produces same output
- **Validates**: 47 actions present, no duplicates, all domains covered
- **Fails CI** if: action count != 47, action names not StringName-valid, etc.

#### Release-build stripping (Debug actions)

- 4 Debug actions: `debug_toggle_god`, `debug_warp`, `debug_fill_codex`, `debug_damage`
- Per `player-input.md` G1: "Debug actions are stripped from release builds"
- Implementation: `tools/strip_debug_actions.py` runs at release build time
  - Reads `input-bindings.yaml`, filters out `is_debug: true`
  - Writes a `project.godot.release` file with 43 actions
  - Release build uses this file
- In dev build, all 47 actions are present
- In release build, only 43 actions (no Debug)

#### Hot-swap behavior (SDL3 backend, 4.5+ HIGH RISK)

- When a gamepad is unplugged:
  - Godot's SDL3 backend (4.5+) clears device bindings
  - InputBus receives `Input.joy_connection_changed` signal
  - HUD's gamepad indicator (per `player-input.md` UI-4) updates to show "disconnected"
  - Battle continues; player can switch to keyboard/mouse
- When a new gamepad is plugged:
  - SDL3 backend assigns a new device ID
  - InputBus re-resolves gamepad bindings automatically (per Godot 4.5+ behavior)
  - HUD's gamepad indicator shows the new device
- This is a Godot 4.5+ behavior; pre-4.5 may differ. **Verify at first use site.**

#### Visual focus vs action dispatch (4.6 dual-focus)

- 4.6 separates two concepts:
  1. **Action dispatch**: `Input.is_action_just_pressed("attack_primary")` — works for any input device
  2. **Visual focus**: which Control has `gui_focus_owner` — only affects keyboard/gamepad UI navigation
- Per `architecture.md` §4a HIGH RISK: "Our input map is 47 actions; visual focus for HUD must be tested with both input methods."
- Per `player-input.md` C-R6: "When a UI element has `focus_mode = FOCUS_ALL` and is the current `viewport.gui_focus_owner`, the SPACE / ENTER / GAMEPAD-A actions mean 'confirm this element' — not the action's state-level meaning."
- **Both work together**: action dispatch + visual focus routing. The dispatch is "did the player press A?"; the focus routing is "where does the press go?".

#### Modifiers (per player-input.md E6 / Blk #3 fix)

- Godot 4.4+ uses `InputEventKey` fields: `shift_pressed`, `ctrl_pressed`, `alt_pressed`, `meta_pressed`
- `player-input.md` (revised) uses `shift_pressed: true` for `flee_battle` (Shift+Esc), `menu_tab_prev` (Shift+Tab), `quit_to_title` (Ctrl+Q)
- YAML supports `modifiers: [SHIFT, CTRL, ALT, META]` as a list
- Sync step converts to `InputEventKey` fields

#### Performance

- `InputMap.get_actions()` at `_ready()` is O(47) = negligible
- `Input.is_action_pressed()` per frame for 47 actions is O(47) = ~0.01ms (well under 0.1ms budget)
- SDL3 hot-swap signal fires once per device change; we listen and emit our own `device_changed` signal

#### What we DON'T do (per MVP scope)

- No rebinding UI (per player-input.md C-R7)
- No per-player binding profile
- No save bindings (bindings are the same for every player)
- No vibration/rumble spec (per player-input.md OQ-2, deferred to Audio GDD)
- No touch input (per technical-preferences.md: "Touch Support: None")

## Alternatives Considered

### Alternative 1: Hardcode bindings in project.godot directly

- **Description**: Edit `project.godot [input]` section by hand; YAML doesn't exist
- **Pros**: Simplest; one file; no sync step
- **Cons**: No source of truth outside Godot; can't diff or review well; easy to introduce typos
- **Estimated Effort**: -50% initial, +200% maintenance
- **Rejection Reason**: We need a single source of truth that lives outside the binary `.tres`/`.godot` files for code review and CI validation.

### Alternative 2: Multiple YAML files (per domain)

- **Description**: `design/registry/input-bindings/exploration.yaml`, `battle_combat.yaml`, etc.
- **Pros**: Modularity; can version per-domain
- **Cons**: 8 files for 47 actions; harder to get the "all 47 in one view"; cross-domain binding (Shift+Q = cycle_weapons + pause) becomes confusing
- **Estimated Effort**: +30% boilerplate, -20% discoverability
- **Rejection Reason**: For 47 actions, one YAML is simpler.

### Alternative 3: Godot `.tres` InputEvent resources

- **Description**: Each binding is a separate `.tres` file with `InputEvent` resource
- **Pros**: Visual editing in Godot Inspector
- **Cons**: 47 files (or more) for 47 actions; can't be reviewed in code review; binary-ish
- **Estimated Effort**: +500% file count, -100% reviewability
- **Rejection Reason**: YAML is the established convention; `.tres` doesn't add value here

### Alternative 4: GDScript bindings (no project.godot, no YAML)

- **Description**: In InputBus, hard-code action mapping in GDScript at `_ready()`
- **Pros**: No project.godot needed; pure code
- **Cons**: Bypasses Godot's InputMap system; can't use `Input.is_action_pressed()`; doesn't follow Godot idiom
- **Estimated Effort**: -30% initial, +500% rewrite cost later
- **Rejection Reason**: Godot's `InputMap` is the engine's idiom. Bypassing it is anti-pattern.

## Consequences

### Positive

- **Single source of truth** — YAML is canonical, code review-able
- **CI enforces closed set** — 47 actions, linter ensures no drift
- **Deterministic** — same bindings for every player (no per-player rebinding for MVP)
- **Documentation** — YAML serves as both config and human-readable spec
- **Hot-swap support** — Godot 4.5+ SDL3 backend handles device changes
- **Stripping in release** — 4 Debug actions excluded from release build
- **Performance** — O(47) per frame, negligible

### Negative

- **Sync step complexity** — `tools/sync_input_bindings.py` must be maintained
- **HIGH RISK** — SDL3 backend behavior on 4.5+ is unverified; first use site must pair with `godot-specialist`
- **Two dual-focus** — action dispatch ≠ visual focus; designers must think in both
- **No rebinding UI** — accessibility deferred (per player-input.md OQ-1)
- **YAML → Godot conversion** — modifiers, axes, gamepad buttons must be correctly translated

### Neutral

- 47 actions is a deliberate number; could be more or fewer in different design iterations
- `pause` and `pause_battle` are separate actions (per Blk #1 fix); could merge
- `cycle_weapons` keyboard binding changed from Q to Shift+Q (per Blk #1 fix); could be reverted
